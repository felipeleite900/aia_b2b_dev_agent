  ## Project Context Analysis

  ### Requirements Overview

  **Functional Requirements:**
  18 FRs across 7 feature groups forming two primary subsystems:

  1. **Data Pipeline Subsystem** (FR-1 to FR-3, FR-13 to FR-15): Daily batch
  pipeline transforming Netscout probe data in BigQuery into curated analytics
  tables. Includes incremental processing, idempotent refreshes, pipeline
  observability, and data quality checks (freshness, coverage, volume anomaly
  detection). Orchestrated via Cloud Workflows on a daily schedule.

  2. **Web Application Subsystem** (FR-4 to FR-12, FR-16 to FR-18):
  Purpose-built internal web app serving quality KPIs, usage analytics,
  comparative carrier rankings, and trend visibility. Authenticated via Google
  IAP. Reads from pre-computed materialized views for sub-second interactivity.
  Deployed on bi-layer Cloud Run.

  **Non-Functional Requirements:**
  - Performance: Pre-computed aggregations, <3s page load, <500ms interactions,
  pipeline completes within 2 hours
  - Reliability: No data corruption on failure, retry on transient errors, last
  successful refresh always available
  - Observability: Structured pipeline logs, application request metrics, data
  quality metrics in UI
  - Data residency: northamerica-northeast1 / northamerica-northeast2 only
  - Security: No PII, Google IAP authentication, <10 concurrent users

  **Scale & Complexity:**
  - Primary domain: Data pipeline + internal web application
  - Complexity level: Medium
  - Data surface: ~200 countries, ~487 partner carriers, daily refresh cadence
  - User base: ~10 concurrent users (carrier steering team)
  - Estimated architectural components: 5-7 (pipeline orchestration, BigQuery
  data model, web API, web UI, authentication layer, monitoring/alerting, data
  quality checks)

  ### Technical Constraints & Dependencies

  - **Netscout probe data in BigQuery**: Primary data source. Schema validation
  is the first implementation task (OQ-1)
  - **BigQuery Enterprise Datahub**: If source data is via `ent_*` tables,
  access PR to `telus/tf-infra-cio-datahub-bilayer` must be merged before
  pipeline development (source: infrastructure/bigquery-enterprise-datahub.md)
  - **Cloud Workflows**: Daily orchestration per bi-layer patterns (source:
  infrastructure/cloud-workflows-orchestration.md)
  - **Bi-layer Cloud Run**: Web app hosting. Sync ≤5s profile fits bi-layer
  (source: infrastructure/bi-layer-overview.md)
  - **Google IAP**: Authentication — standard internal-tool mechanism
  - **Canadian data residency**: All GCP resources in
  northamerica-northeast1/northeast2 (source:
  governance/canadian-data-sovereignty.md)
  - **Composite Quality Score**: V1 default equal-weight average across
  normalized KPIs (0-100). Architecturally critical — drives ranking,
  comparison, and degradation features

  ### Cross-Cutting Concerns Identified

  - **Data quality as a first-class concern**: Quality checks (freshness,
  coverage, anomaly detection) run alongside the pipeline and surface in the web
   app — not a separate monitoring system
  - **Pre-computation strategy**: The gap between raw Netscout data and
  sub-second UI responses requires a materialized view / aggregation layer in
  BigQuery
  - **Composite score computation**: Must be computed during the pipeline phase
  and stored, not calculated at query time — it's consumed by ranking,
  comparison, and degradation detection
  - **Idempotent pipeline design**: Re-running the same day's pipeline must
  produce identical results (FR-1) — influences how intermediate tables are
  written
  - **Backfill mode**: Full 90-day initial load capability required alongside
  daily incremental processing (FR-2)
  - **Degradation detection logic**: Direction-aware threshold checks per KPI
  type — must be baked into the pipeline, not computed in the UI

  Given your basic frontend experience and the need for minimal interactivity

  ---
  Starter Options for Report Generation:

  Option A: Python + Plotly (Recommended)
  - Plotly generates self-contained interactive HTML files — charts with hover,
  zoom, filter built in
  - No JavaScript knowledge needed — you write Python, it outputs HTML
  - Handles tables, trend charts, bar charts, heatmaps natively
  - Single .html file with everything embedded (data + charts + CSS)

  Option B: Python + Jinja2 templates + Chart.js
  - More control over layout and styling
  - Requires some HTML/CSS/JS knowledge
  - Lighter-weight output files
  - More effort to build but more customizable

  Option C: Python + Streamlit (export to HTML)
  - Fastest to prototype but limited static export capabilities
  - Better suited for a live dashboard, not a file-to-share workflow

  For a basic-experience team generating shareable HTML files with interactive
  charts and tables, Option A (Plotly) fits best. You stay entirely in Python,
  and Plotly's plotly.express + dash_table or raw plotly.graph_objects handle
  the chart interactivity out of the box.

  Architectural Boundaries

  Data Boundary — The crt_mv_* Contract:
  The curated materialized views are the single interface between the pipeline
  (SQL) and the report generator (Python). The Python code never reads stg_* or
  raw_* tables. If a new data point is needed in the report, it must first
  appear in a crt_mv_* view.

  Pipeline Boundary — Stored Procs are Independent:
  Each sp_* stored proc reads from one layer and writes to the next. No proc
  reads and writes within the same layer. Execution order is enforced by Cloud
  Workflows, not by SQL dependencies.

  ent_* (raw) → sp_refresh_carrier_kpis → stg_carrier_kpi_daily
                                           ↓
             sp_compute_composite_scores → stg_carrier_composite_daily
                                           ↓
                sp_detect_degradation → stg_carrier_degradation_daily
                                           ↓
               sp_run_quality_checks → stg_data_quality_log
                                           ↓
                    sp_curate_output → crt_mv_* (curated views)
                                           ↓
                Cloud Run Job (Python) → roaming-intelligence-{date}.html → GCS

  Report Boundary — Section Independence:
  Each section module (sections/*.py) receives a DataFrame from bq_client.py and
   returns a Plotly figure or HTML string. Sections don't call each other.
  builder.py orchestrates assembly.

  0-100" but doesn't specify how. Min-max over the 90-day window? Percentile
  rank? Fixed reference values? This affects composite score comparability
  across days. Recommendation: Define during implementation of
  sp_compute_composite_scores — use min-max normalization over the trailing
  90-day window per KPI, recalculated daily.

  document column mapping as an addendum to this architecture once validated.
