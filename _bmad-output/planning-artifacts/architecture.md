# Architecture Decision Document

_This document builds collaboratively through step-by-step discovery. Sections are appended as we work through each architectural decision together._

## Project Context Analysis

### Requirements Overview

**Functional Requirements:**
18 FRs across 7 feature groups forming two primary subsystems:

1. **Data Pipeline Subsystem** (FR-1 to FR-3, FR-13 to FR-15): Daily batch pipeline transforming Netscout probe data in BigQuery into curated analytics tables. Includes incremental processing, idempotent refreshes, pipeline observability, and data quality checks (freshness, coverage, volume anomaly detection). Orchestrated via Cloud Workflows on a daily schedule.

2. **Web Application Subsystem** (FR-4 to FR-12, FR-16 to FR-18): Purpose-built internal web app serving quality KPIs, usage analytics, comparative carrier rankings, and trend visibility. Authenticated via Google IAP. Reads from pre-computed materialized views for sub-second interactivity. Deployed on bi-layer Cloud Run.

**Non-Functional Requirements:**
- Performance: Pre-computed aggregations, <3s page load, <500ms interactions, pipeline completes within 2 hours
- Reliability: No data corruption on failure, retry on transient errors, last successful refresh always available
- Observability: Structured pipeline logs, application request metrics, data quality metrics in UI
- Data residency: northamerica-northeast1 / northamerica-northeast2 only
- Security: No PII, Google IAP authentication, <10 concurrent users

**Scale & Complexity:**
- Primary domain: Data pipeline + internal web application
- Complexity level: Medium
- Data surface: ~200 countries, ~487 partner carriers, daily refresh cadence
- User base: ~10 concurrent users (carrier steering team)
- Estimated architectural components: 5-7 (pipeline orchestration, BigQuery data model, web API, web UI, authentication layer, monitoring/alerting, data quality checks)

### Technical Constraints & Dependencies

- **Netscout probe data in BigQuery**: Primary data source. Schema validation is the first implementation task (OQ-1)
- **BigQuery Enterprise Datahub**: If source data is via `ent_*` tables, access PR to `telus/tf-infra-cio-datahub-bilayer` must be merged before pipeline development (source: infrastructure/bigquery-enterprise-datahub.md)
- **Cloud Workflows**: Daily orchestration per bi-layer patterns (source: infrastructure/cloud-workflows-orchestration.md)
- **Bi-layer Cloud Run**: Web app hosting. Sync ≤5s profile fits bi-layer (source: infrastructure/bi-layer-overview.md)
- **Google IAP**: Authentication — standard internal-tool mechanism
- **Canadian data residency**: All GCP resources in northamerica-northeast1/northeast2 (source: governance/canadian-data-sovereignty.md)
- **Composite Quality Score**: V1 default equal-weight average across normalized KPIs (0-100). Architecturally critical — drives ranking, comparison, and degradation features

### Cross-Cutting Concerns Identified

- **Data quality as a first-class concern**: Quality checks (freshness, coverage, anomaly detection) run alongside the pipeline and surface in the web app — not a separate monitoring system
- **Pre-computation strategy**: The gap between raw Netscout data and sub-second UI responses requires a materialized view / aggregation layer in BigQuery
- **Composite score computation**: Must be computed during the pipeline phase and stored, not calculated at query time — it's consumed by ranking, comparison, and degradation detection
- **Idempotent pipeline design**: Re-running the same day's pipeline must produce identical results (FR-1) — influences how intermediate tables are written
- **Backfill mode**: Full 90-day initial load capability required alongside daily incremental processing (FR-2)
- **Degradation detection logic**: Direction-aware threshold checks per KPI type — must be baked into the pipeline, not computed in the UI

## Starter Template Evaluation

### Architecture Pivot: Static HTML Reports

The PRD's Feature 4.7 (deployed web application with Cloud Run, IAP, real-time interactivity) is replaced by **static HTML reports generated by the pipeline**. This eliminates the web application subsystem entirely and consolidates the project into a single subsystem: a BigQuery data pipeline that produces shareable HTML output.

**Rationale:** The steering team's workflow is better served by a daily HTML report they can open, share, and review offline. No deployment infrastructure, no authentication layer, no ongoing hosting. Distribution is manual (file sharing).

**Impact on PRD requirements:**
- FR-16 (authenticated access): Replaced by file-level access control (who receives the file)
- FR-17 (responsive data views): Replaced by Plotly's built-in chart interactivity (hover, zoom, filter)
- FR-18 (filter and search): Simplified to Plotly's native filtering + HTML-embedded table sorting
- FR-4 through FR-12 (quality views, rankings, trends): Delivered as sections within the HTML report
- All pipeline FRs (FR-1 to FR-3, FR-13 to FR-15): Unchanged

## Core Architectural Decisions

### Decision Priority Analysis

**Critical Decisions (Block Implementation):**
- BigQuery data model: three-layer (raw → work/staging → curated)
- Partitioning: daily date-partitioned by `refresh_date`
- Composite Quality Score: stored proc computation, persisted as column
- Pipeline composition: monolithic Cloud Workflows stack
- Report generation: Cloud Run Job as final pipeline step

**Important Decisions (Shape Architecture):**
- Report format: single self-contained HTML file
- Output location: GCS bucket, date-versioned
- Pipeline notifications: gchat with report link on success/failure
- Environment strategy: both `bi-stg` and `bi-srv` active, separate GCP projects with segregated bucket names

**Deferred Decisions (Post-MVP):**
- Alerting/notification beyond gchat (email distribution, scheduled sends)
- Composite Quality Score weight customization UI
- Sub-daily refresh cadence

### Data Architecture

**Data Model: Three-Layer BigQuery Pipeline**
- **Layer 1 — Raw source**: Netscout probe data in BigQuery (read-only, owned by upstream). If served via Datahub `ent_*` tables, access bound via `telus/tf-infra-cio-datahub-bilayer` JSON config. (source: infrastructure/bigquery-enterprise-datahub.md §4)
- **Layer 2 — Work/staging tables**: Intermediate results from each pipeline step (normalized KPIs, per-carrier-per-country aggregations). Used for debugging, reprocessing, and step isolation. Written by stored procs during pipeline execution.
- **Layer 3 — Curated output**: Final materialized views consumed by the report generator. Includes composite quality scores, degradation flags, data quality indicators, usage analytics. This is the contract between the pipeline and the report.

**Partitioning Strategy:** Daily date-partitioned by `refresh_date`. Each pipeline run writes to a single partition. Rerunning the same day overwrites the partition — idempotency by design (FR-1). Backfill overwrites specific date partitions without affecting other dates.

**Composite Quality Score:** Computed in a BigQuery stored procedure. Equal-weight average of normalized KPIs (each KPI scaled 0-100). Stored as a column (`composite_quality_score`) in the curated carrier-level table. Single computation point consumed by ranking (FR-9), comparison (FR-10), and degradation detection (FR-12).

**Historical Retention:** 90 days in curated tables. Aligns with the maximum trend window (FR-11). Older partitions are dropped on a rolling basis. Raw source data retention is upstream's responsibility.

### Pipeline Architecture

**Composition Pattern: Monolithic Stack**
Single `Pulumi.yaml` with inline `sourceContents` defining the Cloud Workflows step chain. Appropriate for 1 workflow with 5-6 sequential steps. (source: infrastructure/cloud-workflows-orchestration.md §7)

**Pipeline Step Chain:**
1. `step1_refresh_data` — Incremental load: BQ stored proc reads new/changed raw data, writes to work tables
2. `step2_compute_kpis` — Normalize individual KPIs (0-100 scale), compute composite quality score, write to work tables
3. `step3_detect_degradation` — Direction-aware threshold checks per KPI over 7-day trailing window, write degradation flags
4. `step4_quality_checks` — Freshness validation, coverage diff vs. prior refresh, volume anomaly detection (50% of 14-day rolling average)
5. `step5_curate` — Write final curated tables/materialized views from work tables
6. `step6_generate_report` — Trigger Cloud Run Job: Python container queries curated tables, generates HTML via Plotly + Jinja2, writes to GCS bucket

**Report Generation Step:** Cloud Run Job triggered as the final Cloud Workflows step. Clean separation: SQL handles all data transformation (steps 1-5), Python handles presentation (step 6). The Cloud Run container image lives in `images/roaming-intelligence/`.

**Error Handling:** Fail-fast. If any step fails, the workflow stops and sends a Google Chat alert via the shared `aia_gchat_workflow` library workflow. (source: infrastructure/cloud-workflows-orchestration.md §8.3) Last successful curated data and report remain available — no corruption on failure (FR-1 reliability NFR).

**Schedule:** `0 6 * * 1-5` — weekdays at 06:00 ET. `time_zone: America/Toronto`. Active in both `bi-stg` and `bi-srv`. (source: infrastructure/cloud-workflows-orchestration.md §6)

### Report Architecture

**Format:** Single self-contained HTML file. All charts (Plotly JS), data, and styling embedded. No external CDN dependencies. Shareable via email, Slack, or direct GCS link.

**Report Sections:**
1. **Executive Summary** — Data freshness timestamp, country/carrier coverage stats, degradation alerts, data quality flags
2. **Country Overview** — Sortable table of all countries with aggregate quality indicator
3. **Carrier Rankings by Country** — Per-country: carriers ranked by composite score, individual KPIs visible, steered-to carrier flagged with `is_steered` indicator
4. **Trend Charts** — Plotly interactive line charts per country: 7/14/30/90-day KPI trends per carrier (hover, zoom, filter built in)
5. **Usage Analytics** — Traffic volume, session count, subscriber count per carrier; traffic share percentages; <1% carriers grouped under "Other"
6. **Data Quality** — Freshness check result, coverage validation (missing countries/carriers vs. prior refresh), volume anomaly flags

**Output Location:** GCS bucket, date-versioned path: `reports/{YYYY-MM-DD}/roaming-intelligence.html`. Separate buckets per environment (segregated names across the `bi-stg` and `bi-srv` GCP projects).

### Infrastructure & Deployment

**Bi-layer Monorepo Structure:**

**Monitoring & Alerting:**
- Pipeline failure: gchat notification via `aia_gchat_workflow` with error context
- Pipeline success: gchat notification with direct GCS link to the generated report
- Pipeline observability: structured logs per run (start time, end time, rows processed, countries/carriers covered, success/failure) per FR-3

### Security & Access

**BigQuery Access:** Service account bound via `telus/tf-infra-cio-datahub-bilayer` JSON config. Both `bi-stg` and `bi-srv` SAs get `roles/bigquery.dataViewer` on source datasets, `roles/bigquery.dataEditor` on project-owned datasets. (source: infrastructure/bigquery-enterprise-datahub.md §4)

**GCS Report Bucket:** IAM-scoped. Pipeline service account: write (`roles/storage.objectCreator`). Steering team Google group: read (`roles/storage.objectViewer`). No public access.

**Data Classification:** No PII in the pipeline. Aggregated network performance metrics at carrier/country level only. Classification: Internal. (source: governance/telus-trust-model.md)

1. Fork bi-layer stacks catalog templates, set up project structure

## Implementation Patterns & Consistency Rules

### Data Warehouse Layer Convention

All BigQuery objects carry a layer prefix indicating their position in the data pipeline:

| Prefix | Layer | Purpose |
|---|---|---|
| `raw_` | Raw | Source data (upstream Netscout / Datahub `ent_*` — read-only, naming is Datahub's) |
| `stg_` | Staging | Intermediate pipeline outputs — work tables written by stored procs |
| `crt_` | Curated | Final output contract consumed by the report generator |

### BigQuery Naming Conventions

**Object Type Prefixes** (per SSOT: source: engineering/sql-best-practices.md §6):
- `sp_` — stored procedures
- `vw_` — views
- `mv_` — materialized views
- `bq_` — ad-hoc query scripts

**Combined naming pattern**: `{layer}_{type_prefix}_{descriptor}` for views/MVs; `{layer}_{descriptor}` for tables; `sp_{descriptor}` for stored procs (no layer prefix — procs operate across layers).

**Dataset:** `roaming_intelligence`

**Table and View Names:**

| Layer | Object | Name |
|---|---|---|
| raw | Source tables | `ent_*` (upstream Datahub, read-only — naming is theirs) |
| stg | Daily KPI table | `stg_carrier_kpi_daily` |
| stg | Composite score table | `stg_carrier_composite_daily` |
| stg | Degradation flags table | `stg_carrier_degradation_daily` |
| stg | Quality check log | `stg_data_quality_log` |
| crt | Carrier quality summary | `crt_mv_carrier_quality_summary` |

**Stored Procedure Names:**

### Python Code Conventions

- **Naming**: `snake_case` for all Python identifiers (PEP 8) (source: engineering/python-clean-code.md)
- **Logging**: `structlog` for structured logging with correlation IDs (source: engineering/python-logging.md)
- **Type hints**: Required on all function signatures
- **Docstrings**: Google-style (source: engineering/python-documentation.md)
- **Report functions**: One function per report section (e.g., `generate_country_overview()`, `generate_carrier_rankings()`)
- **Plotly API**: Use `plotly.graph_objects` (not `plotly.express`) for consistency and fine-grained control
- **Jinja2**: Base template defines HTML structure; section functions inject Plotly HTML into template slots

### Process Patterns

**Idempotent Writes:** All stored procs use `DELETE` + `INSERT` on the target partition (not `MERGE`). Simpler, guarantees clean partition state on rerun.

**Pipeline Logging:** Each stored proc logs to `stg_data_quality_log`: step name, start time, end time, row count, status. This table is the observability contract (FR-3).

**Error Propagation:** Cloud Workflows step failures propagate the BigQuery error message to the gchat notification via `aia_gchat_workflow`.

**Date Handling:** All dates as BigQuery `DATE` type, ISO 8601 (`YYYY-MM-DD`). All timestamps as `TIMESTAMP` type, UTC. Report filename includes date: `roaming-intelligence-{YYYY-MM-DD}.html`.

### Enforcement Guidelines

**All AI Agents MUST:**
- Use the layer prefix convention (`stg_`, `crt_`) for every new table or view
- Follow the column naming standard — no ad-hoc column names outside this schema without updating this document
- Use `DELETE` + `INSERT` for idempotent writes, never `MERGE` or `INSERT` without clearing the partition
- Log every stored proc execution to `stg_data_quality_log`
- Use `plotly.graph_objects` (not `plotly.express`) for all chart generation
- Include type hints on all Python function signatures

**Anti-Patterns:**
- Creating tables without a layer prefix
- Using `camelCase` in BigQuery column names
- Writing to curated (`crt_*`) tables directly from a stored proc that reads raw — always go through staging
- Hardcoding GCP project IDs or bucket names — use Jinja `PROJECT_TYPE` / `environment.j2` variables
- Mixing Plotly Express and Graph Objects in the same codebase

### Architectural Boundaries

**Data Boundary — The `crt_mv_*` Contract:**
The curated materialized views are the single interface between the pipeline (SQL) and the report generator (Python). The Python code never reads `stg_*` or `raw_*` tables. If a new data point is needed in the report, it must first appear in a `crt_mv_*` view.

**Pipeline Boundary — Stored Procs are Independent:**
Each `sp_*` stored proc reads from one layer and writes to the next. No proc reads and writes within the same layer. Execution order is enforced by Cloud Workflows, not by SQL dependencies.

### Data Flow

```
ent_* (raw) → sp_refresh_carrier_kpis → stg_carrier_kpi_daily
                                         ↓
           sp_compute_composite_scores → stg_carrier_composite_daily
                                         ↓
              sp_detect_degradation → stg_carrier_degradation_daily
                                         ↓
             sp_run_quality_checks → stg_data_quality_log
                                         ↓
                  sp_curate_output → crt_mv_* (curated views)
                                         ↓
              Cloud Run Job (Python) → roaming-intelligence-{date}.html → GCS
```

**Report Boundary — Section Independence:**
Each section module (`sections/*.py`) receives a DataFrame from `bq_client.py` and returns a Plotly figure or HTML string. Sections don't call each other. `builder.py` orchestrates assembly.
