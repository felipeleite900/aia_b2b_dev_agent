# International Roaming Intelligence - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for International Roaming Intelligence, decomposing the requirements from the PRD and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR-1: The system refreshes all Quality KPIs and usage metrics from Netscout probe data in BigQuery on a daily schedule. Curated analytics tables updated by 08:00 ET daily. Each refresh covers all countries and Partner Carriers. The refresh is idempotent.

FR-2: The pipeline processes only new or changed data since the last successful refresh. Pipeline execution time scales with daily volume. A full backfill mode exists for initial load and recovery.

FR-3: The pipeline emits structured logs and metrics for each run. Each run records: start time, end time, rows processed, countries covered, carriers covered, success/failure status. Failed runs logged with error context.

FR-4: The system displays a list of countries with an aggregate quality indicator for each. All countries with roaming data in the latest refresh are listed.

FR-5: For a selected country, the system displays all Partner Carriers ranked by Composite Quality Score with individual Quality KPIs visible. KPIs: Latency (ms), Throughput (kbps), Packet Loss (%), Session Success Rate (%). Steered-to carrier identified with distinct label.

FR-6: The system allows drilling into a specific Partner Carrier to see detailed KPI values, last 30 days of daily KPI values, and metadata (carrier name, MCC/MNC, country).

FR-7: The system displays usage metrics alongside quality metrics: traffic volume (MB), session count, unique subscriber count. Usage metrics from the same Refresh Cycle as Quality KPIs.

FR-8: The system shows each Partner Carrier's share of total roaming traffic within a country as percentage. Carriers with <1% share grouped under "Other".

FR-9: The system supports ranking Partner Carriers by individual KPIs, Composite Quality Score, or usage volume. Default sort by Composite Quality Score.

FR-10: The system allows selecting two or more Partner Carriers for direct side-by-side comparison. KPIs in aligned columns with better values indicated. Difference >10% flagged as significant.

FR-11: The system displays Quality KPI trends over configurable time windows. Default: 30-day. Configurable: 7, 14, 30, 90 days. Trend lines per Partner Carrier within a country.

FR-12: The system flags carriers whose quality has degraded beyond a threshold over a 7-day trailing window. Direction-aware per KPI. Default threshold 20%, configurable.

FR-13: The system validates data freshness. Displays timestamp of last successful Refresh Cycle. Staleness warning if data older than 36 hours.

FR-14: The system checks expected countries and carriers are present in each refresh. Missing entities flagged. Coverage summary visible.

FR-15: The system flags statistically significant drops in data volumes. Daily carrier-country pair with traffic below 50% of 14-day rolling average is flagged.

FR-16: The application is accessible only to authorized users. [ARCHITECTURE PIVOT: GCS IAM-scoped access replaces Google IAP web app authentication.]

FR-17: Responsive data views with interactivity. [ARCHITECTURE PIVOT: Plotly built-in interactivity (hover, zoom, filter) replaces web app sub-second interactivity.]

FR-18: Filter and search across countries and carriers. [ARCHITECTURE PIVOT: Simplified to Plotly native filtering + HTML table sorting.]

### NonFunctional Requirements

NFR-1: Performance — Pre-computed aggregations ensure sub-second query response. Pipeline completes daily refresh within 2 hours of data availability.

NFR-2: Reliability — Pipeline failures do not corrupt existing data; last successful refresh remains available. Retry logic on transient failures (BigQuery slot contention, network hiccups).

NFR-3: Observability — Structured logging for pipeline runs (structlog). Application-level request metrics. Data quality metrics exposed in the report.

NFR-4: Data Residency — All data processing and storage in northamerica-northeast1 or northamerica-northeast2 per TELUS Canadian data sovereignty requirements. (source: governance/canadian-data-sovereignty.md)

NFR-5: Security — No PII in the pipeline or report output. Network performance data only. Access scoped to the steering team via GCS IAM.

### Additional Requirements

- **Architecture Pivot**: Static HTML reports generated by the pipeline replace the deployed web application. Single self-contained HTML file with Plotly charts. No Cloud Run service deployment, no IAP authentication layer.
- **Starter Template**: Fork bi-layer stacks catalog templates (`common_patterns/scheduled_bq_multi_step_stored_proc`, `big_query/*`) into the team's fork as the first implementation task. (source: infrastructure/bi-layer-overview.md §5)
- **Three-Layer BigQuery Data Model**: raw (ent_* upstream) → stg (staging/work tables) → crt (curated materialized views). Strict layer discipline enforced.
- **Pipeline Composition**: Monolithic Cloud Workflows stack with 6 sequential steps: data refresh → KPI computation → degradation detection → quality checks → curate output → generate report.
- **Report Generation**: Cloud Run Job as final pipeline step. Python + Plotly (graph_objects only) + Jinja2. Output to GCS bucket, date-versioned: `reports/{YYYY-MM-DD}/roaming-intelligence.html`.
- **Naming Conventions**: Layer prefixes (stg_, crt_), object type prefixes (sp_, vw_, mv_), snake_case columns with kpi_/norm_ domain prefixes.
- **Idempotent Writes**: All stored procs use DELETE + INSERT on target partition, never MERGE.
- **Pipeline Logging**: Each stored proc logs to stg_data_quality_log (step name, start/end time, row count, status).
- **Error Handling**: Fail-fast with gchat notification via aia_gchat_workflow on failure. Success notification with GCS report link.
- **Schedule**: Weekdays 06:00 ET (0 6 * * 1-5), time_zone: America/Toronto. Active in both bi-stg and bi-srv.
- **Environment Strategy**: Both bi-stg and bi-srv active. Separate GCS buckets per env. Jinja PROJECT_TYPE switches env-specific config.
- **Schema Validation**: OQ-1 (Netscout BigQuery schema) is the highest-priority item — unblocks all pipeline work.
- **Datahub Access**: If source data is via ent_* tables, access PR to telus/tf-infra-cio-datahub-bilayer must be merged before pipeline development. (source: infrastructure/bigquery-enterprise-datahub.md)

N/A — No UX Design document. This is a data pipeline project with static HTML report output.

## Epic 2: Daily Quality Intelligence Pipeline

Build all data transformation logic — from raw Netscout data through KPI normalization, composite scoring, degradation detection, and data quality checks to curated output views. After this epic, the crt_mv_* data contract is fulfilled and curated tables can be queried directly.

### Story 2.1: Build KPI Refresh Stored Procedure

As a carrier steering analyst,
I want raw Netscout probe data incrementally loaded and normalized into daily KPI values per carrier per country,
So that I have up-to-date quality metrics to base steering decisions on.

**Acceptance Criteria:**

**Given** Netscout source data exists in BigQuery for the current date
**When** `sp_refresh_carrier_kpis` executes
**Then** `stg_carrier_kpi_daily` is populated with one row per carrier per country per day
**And** raw KPI columns are populated: `kpi_latency_ms`, `kpi_throughput_kbps`, `kpi_packet_loss_pct`, `kpi_session_success_pct`
**And** normalized KPI columns (0-100 scale) are populated: `norm_latency` (inverted), `norm_throughput`, `norm_packet_loss` (inverted), `norm_session_success`
**And** usage metrics are populated: `traffic_volume_mb`, `session_count`, `subscriber_count`
**And** metadata columns are populated: `country_name`, `country_code`, `carrier_name`, `mcc`, `mnc`, `is_steered`
**And** the proc uses DELETE + INSERT on the `refresh_date` partition for idempotency
**And** only new/changed data since the last refresh is processed (incremental)
**And** execution is logged to `stg_data_quality_log` with start time, end time, row count, status

### Story 2.2: Build Composite Quality Score Stored Procedure

As a carrier steering analyst,
I want Partner Carriers scored with a composite quality metric that combines all individual KPIs,
So that I can rank carriers at a glance and compare them objectively.

**Acceptance Criteria:**

**Given** `stg_carrier_kpi_daily` is populated with normalized KPIs for the current date
**When** `sp_compute_composite_scores` executes
**Then** `stg_carrier_composite_daily` is populated with `composite_quality_score` as the equal-weight average of `norm_latency`, `norm_throughput`, `norm_packet_loss`, `norm_session_success`
**And** the score is a FLOAT64 between 0 and 100
**And** rows with missing KPIs compute the average over available KPIs only (graceful handling)
**And** the proc uses DELETE + INSERT on the `refresh_date` partition
**And** execution is logged to `stg_data_quality_log`

### Story 2.3: Build Degradation Detection Stored Procedure

As a carrier steering analyst,
I want to be alerted when a carrier's quality drops significantly over a trailing window,
So that I can react to degradation in days, not weeks.

**Acceptance Criteria:**

**Given** `stg_carrier_kpi_daily` contains at least 7 days of historical data for a carrier
**When** `sp_detect_degradation` executes
**Then** `stg_carrier_degradation_daily` is populated with `degradation_flag` (BOOL) and `degradation_details` (JSON) per carrier per country
**And** degradation is direction-aware: latency increase > 20% = degradation; throughput decrease > 20% = degradation; packet loss increase > 20% = degradation; session success decrease > 20% = degradation
**And** the threshold is compared against the 7-day trailing average
**And** `degradation_details` JSON includes which KPIs degraded and the percentage change
**And** carriers with fewer than 7 days of data are not flagged (insufficient baseline)
**And** the proc uses DELETE + INSERT on the `refresh_date` partition
**And** execution is logged to `stg_data_quality_log`

### Story 2.4: Build Data Quality Checks Stored Procedure

As a carrier steering analyst,
I want automated validation that the data is fresh, complete, and not anomalous,
So that I can trust the data before making steering decisions.

**Acceptance Criteria:**

**Given** the current pipeline run has completed steps 1-3
**When** `sp_run_quality_checks` executes
**Then** freshness is validated: the latest `refresh_date` is within 36 hours of current time; if stale, a warning record is written to `stg_data_quality_log`
**And** coverage is validated: countries and carriers present in the prior refresh but missing in the current refresh are identified and logged
**And** volume anomaly detection runs: any carrier-country pair with traffic volume below 50% of its 14-day rolling average is flagged
**And** a coverage summary is written to `stg_data_quality_log`: countries covered, carriers covered, missing entities count, anomaly count
**And** the proc uses DELETE + INSERT on the `refresh_date` partition for its output
**And** execution is logged to `stg_data_quality_log`

### Story 2.5: Build Curated Output Layer

As a carrier steering analyst,
I want the final curated views to aggregate all pipeline outputs into a clean data contract,
So that the report generator has a single, reliable interface to read from.

**Acceptance Criteria:**

**Given** all staging tables are populated for the current date
**When** `sp_curate_output` executes
**Then** `crt_mv_carrier_quality_summary` is populated by joining `stg_carrier_kpi_daily`, `stg_carrier_composite_daily`, and `stg_carrier_degradation_daily`
**And** `crt_mv_country_quality_summary` is populated with per-country aggregate quality indicators
**And** `crt_mv_carrier_quality_trend` is populated with the last 90 days of daily KPI values per carrier
**And** `crt_mv_carrier_usage_summary` is populated with traffic volume, session count, subscriber count, and traffic share percentage per carrier per country
**And** carriers with < 1% traffic share are tagged for "Other" grouping
**And** the proc uses DELETE + INSERT on the `refresh_date` partition
**And** execution is logged to `stg_data_quality_log`
