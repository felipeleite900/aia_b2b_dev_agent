  ---
  Epic Design Assessment

  This project has a fully validated architecture with clear direction (static
  HTML reports, monolithic Cloud Workflows, three-layer BQ model). Direction
  changes between epics are unlikely, so fewer, larger epics are appropriate.

  Key observations:
  - The architecture consolidates everything into one subsystem (pipeline +
  report)
  - SQL stored procs are independent files, Python report sections are
  independent files
  - The main shared file (Pulumi.yaml) grows incrementally across setup →
  pipeline → orchestration
  - The user (steering analyst) consumes a single output: the daily HTML report

  ---
  Proposed Epic Structure

  Epic 1: Project Foundation & Data Infrastructure
  Set up the bi-layer project structure, validate the Netscout data source, and
  establish BigQuery resources so the team can start building pipeline logic.
  FRs covered: Enables FR-1, FR-2 (infrastructure prerequisites)

  Epic 2: Daily Quality Intelligence Pipeline
  Build all data transformation logic — from raw Netscout data through KPI
  normalization, composite scoring, degradation detection, and data quality
  checks to curated output views. After this epic, the crt_mv_* data contract is
   fulfilled and curated tables can be queried directly.
  FRs covered: FR-1, FR-2, FR-3, FR-5 (data), FR-7 (data), FR-9 (data), FR-11
  (data), FR-12, FR-13, FR-14, FR-15

  Epic 3: Interactive Report Generation & Distribution
  Build the Python report generator that reads curated views and produces a
  self-contained HTML report with all analytical views — quality rankings, usage
   analytics, trend charts, side-by-side comparison, and data quality
  indicators. After this epic, reports can be generated manually.
  FRs covered: FR-4, FR-5 (presentation), FR-6, FR-7, FR-8, FR-9, FR-10, FR-11
  (presentation), FR-16, FR-17, FR-18

  Epic 4: Pipeline Orchestration & Operational Readiness
  Wire Cloud Workflows to chain all pipeline steps and report generation, set up
   the Cloud Scheduler, configure gchat notifications, run the 90-day historical
   backfill, and validate end-to-end in bi-stg before promoting to bi-srv. After
   this epic, the steering team receives daily automated reports.
  FRs covered: FR-1 (scheduling), FR-3 (end-to-end observability), NFR-1, NFR-2

  Epic 5: Governance & Compliance
  Complete all mandatory governance gates for a data pipeline project: DEP
  Review, DNTL Classification, and De-identification Standards assessment.
  FRs covered: NFR-4, NFR-5, governance gates (source: lifecycle-config.yaml)

  Proposed Stories

  Story 1.1: Set Up Bi-Layer Project Structure
  As a data engineer,
  I want the bi-layer monorepo structure forked from catalog templates with
  stacks/ and images/ pillars,
  So that the team has a standard project foundation to build pipeline and
  report components on.

  Acceptance Criteria:

  Given the team's fork of the bi-layer monorepo exists
  When the project structure is created
  Then stacks/roaming-intelligence/ contains a Pulumi.yaml forked from
  scheduled_bq_multi_step_stored_proc and an environment.j2 with env-specific
  variables (bucket names, schedule, pause status)
  And images/roaming-intelligence/ contains a Dockerfile, pyproject.toml, src/
  directory structure, and templates/ directory per the architecture spec
  And images/roaming-intelligence/sql/ directory exists for stored procedure
  source files
  And all GCP resources are configured for northamerica-northeast1
  And environment.j2 includes Jinja PROJECT_TYPE switches for bi-stg and bi-srv

  ---
  Story 1.2: Validate Netscout BigQuery Schema
  As a data engineer,
  I want the Netscout probe data schema in BigQuery explored and documented,
  So that the team confirms which KPIs are available and can map source columns
  to the target data model.

  Acceptance Criteria:

  Given the Netscout probe data exists in BigQuery (via Datahub ent_* tables or
  direct)
  When the schema is validated
  Then a schema mapping document is produced listing source columns and their
  mapping to target columns (kpi_latency_ms, kpi_throughput_kbps,
  kpi_packet_loss_pct, kpi_session_success_pct, country_name, country_code,
  carrier_name, mcc, mnc)
  And any KPIs not available in the source schema are flagged with mitigation
  options
  And the data grain (country/carrier/day) is confirmed
  And the is_steered field source is identified (programmatic or manual tagging)
  And OQ-1 is resolved with findings documented

  ---
  Story 1.3: Request Datahub Access for Source Data
  As a data engineer,
  I want a PR submitted to telus/tf-infra-cio-datahub-bilayer granting the
  project's service accounts access to the Netscout source datasets,
  So that the pipeline can read source data in both bi-stg and bi-srv
  environments.

  Acceptance Criteria:

  Given the Netscout source datasets are identified from Story 1.2
  When the access PR is submitted
  Then terraform/bi/roaming-intelligence.json lists both bi-stg and bi-srv
  service accounts
  And the enterprise.datasets array includes all required ent_* dataset names
  And the PR targets telus/tf-infra-cio-datahub-bilayer for CODEOWNERS review
  And after merge and Terraform apply, queries against the listed datasets
  succeed from the project's service accounts
  (source: infrastructure/bigquery-enterprise-datahub.md §4-5)

  ---
  Story 1.4: Create BigQuery Dataset and Core Table Definitions
  As a data engineer,
  I want the roaming_intelligence BigQuery dataset and all staging/curated table
   schemas defined in Pulumi IaC,
  So that the pipeline stored procedures have target tables to write to.

  Acceptance Criteria:

  Given the project Pulumi stack exists from Story 1.1
  When the BigQuery resources are provisioned
  Then the roaming_intelligence dataset is created in northamerica-northeast1
  And staging tables are created: stg_carrier_kpi_daily,
  stg_carrier_composite_daily, stg_carrier_degradation_daily,
  stg_data_quality_log
  And curated materialized view placeholders are defined:
  crt_mv_carrier_quality_summary, crt_mv_country_quality_summary,
  crt_mv_carrier_quality_trend, crt_mv_carrier_usage_summary
  And all tables are daily date-partitioned by refresh_date
  And column schemas match the architecture spec (snake_case, kpi_/norm_
  prefixes)
  And IAM bindings grant the pipeline service account roles/bigquery.dataEditor
  on project-owned datasets

  Epic 2: Daily Quality Intelligence Pipeline

  Goal: Build all data transformation logic — from raw Netscout data through KPI
   normalization, composite scoring, degradation detection, and data quality
  checks to curated output views. After this epic, the crt_mv_* data contract is
   fulfilled.

  FRs covered: FR-1, FR-2, FR-3, FR-5 (data), FR-7 (data), FR-9 (data), FR-11
  (data), FR-12, FR-13, FR-14, FR-15

  Relevant architecture decisions:
  - 5 stored procs in sequence + 1 backfill proc
  - DELETE + INSERT for idempotency
  - Each proc logs to stg_data_quality_log
  - Layer discipline: raw → stg → crt
  - Composite score: equal-weight average of normalized KPIs (0-100)
  - Degradation: direction-aware, 7-day window, 20% default threshold

  Proposed Stories

  Story 2.1: Build KPI Refresh Stored Procedure
  As a carrier steering analyst,
  I want raw Netscout probe data incrementally loaded and normalized into daily
  KPI values per carrier per country,
  So that I have up-to-date quality metrics to base steering decisions on.

  Acceptance Criteria:

  Given Netscout source data exists in BigQuery for the current date
  When sp_refresh_carrier_kpis executes
  Then stg_carrier_kpi_daily is populated with one row per carrier per country
  per day
  And raw KPI columns are populated: kpi_latency_ms, kpi_throughput_kbps,
  kpi_packet_loss_pct, kpi_session_success_pct
  And normalized KPI columns (0-100 scale) are populated: norm_latency
  (inverted), norm_throughput, norm_packet_loss (inverted), norm_session_success
  And usage metrics are populated: traffic_volume_mb, session_count,
  subscriber_count
  And metadata columns are populated: country_name, country_code, carrier_name,
  mcc, mnc, is_steered
  And the proc uses DELETE + INSERT on the refresh_date partition for
  idempotency
  And only new/changed data since the last refresh is processed (incremental)
  And execution is logged to stg_data_quality_log with start time, end time, row
   count, status

  ---
  Story 2.2: Build Composite Quality Score Stored Procedure
  As a carrier steering analyst,
  I want Partner Carriers scored with a composite quality metric that combines
  all individual KPIs,
  So that I can rank carriers at a glance and compare them objectively.

  Acceptance Criteria:

  Given stg_carrier_kpi_daily is populated with normalized KPIs for the current
  date
  When sp_compute_composite_scores executes
  Then stg_carrier_composite_daily is populated with composite_quality_score as
  the equal-weight average of norm_latency, norm_throughput, norm_packet_loss,
  norm_session_success
  And the score is a FLOAT64 between 0 and 100
  And rows with missing KPIs compute the average over available KPIs only
  (graceful handling)
  And the proc uses DELETE + INSERT on the refresh_date partition
  And execution is logged to stg_data_quality_log

  ---
  Story 2.3: Build Degradation Detection Stored Procedure
  As a carrier steering analyst,
  I want to be alerted when a carrier's quality drops significantly over a
  trailing window,
  So that I can react to degradation in days, not weeks.

  Acceptance Criteria:

  Given stg_carrier_kpi_daily contains at least 7 days of historical data for a
  carrier
  When sp_detect_degradation executes
  Then stg_carrier_degradation_daily is populated with degradation_flag (BOOL)
  and degradation_details (JSON) per carrier per country
  And degradation is direction-aware: latency increase > 20% = degradation;
  throughput decrease > 20% = degradation; packet loss increase > 20% =
  degradation; session success decrease > 20% = degradation
  And the threshold is compared against the 7-day trailing average
  And degradation_details JSON includes which KPIs degraded and the percentage
  change
  And carriers with fewer than 7 days of data are not flagged (insufficient
  baseline)
  And the proc uses DELETE + INSERT on the refresh_date partition
  And execution is logged to stg_data_quality_log

  ---
  Story 2.4: Build Data Quality Checks Stored Procedure
  As a carrier steering analyst,
  I want automated validation that the data is fresh, complete, and not
  anomalous,
  So that I can trust the data before making steering decisions.

  Acceptance Criteria:

  Given the current pipeline run has completed steps 1-3
  When sp_run_quality_checks executes
  Then freshness is validated: the latest refresh_date is within 36 hours of
  current time; if stale, a warning record is written to stg_data_quality_log
  And coverage is validated: countries and carriers present in the prior refresh
   but missing in the current refresh are identified and logged
  And volume anomaly detection runs: any carrier-country pair with traffic
  volume below 50% of its 14-day rolling average is flagged
  And a coverage summary is written to stg_data_quality_log: countries covered,
  carriers covered, missing entities count, anomaly count
  And the proc uses DELETE + INSERT on the refresh_date partition for its output
  And execution is logged to stg_data_quality_log

  ---
  Story 2.5: Build Curated Output Layer
  As a carrier steering analyst,
  I want the final curated views to aggregate all pipeline outputs into a clean
  data contract,
  So that the report generator has a single, reliable interface to read from.

  Acceptance Criteria:

  Given all staging tables are populated for the current date
  When sp_curate_output executes
  Then crt_mv_carrier_quality_summary is populated by joining
  stg_carrier_kpi_daily, stg_carrier_composite_daily, and
  stg_carrier_degradation_daily
  And crt_mv_country_quality_summary is populated with per-country aggregate
  quality indicators
  And crt_mv_carrier_quality_trend is populated with the last 90 days of daily
  KPI values per carrier
  And crt_mv_carrier_usage_summary is populated with traffic volume, session
  count, subscriber count, and traffic share percentage per carrier per country
  And carriers with < 1% traffic share are tagged for "Other" grouping
  And the proc uses DELETE + INSERT on the refresh_date partition
  And execution is logged to stg_data_quality_log

  ---
  Story 2.6: Build Historical Backfill Procedure
  As a data engineer,
  I want to backfill 90 days of historical data through the full pipeline,
  So that the report has sufficient historical context for trend analysis from
  day one.

  Acceptance Criteria:

  Given the pipeline stored procs (Stories 2.1-2.5) are functional
  When sp_backfill_historical executes with a date range parameter
  Then all pipeline steps run for each date in the range, writing to the correct
   refresh_date partition
  And the backfill is idempotent — rerunning for the same dates produces
  identical results
  And execution completes within a reasonable time for a 90-day range
  And all curated views contain 90 days of historical data after completion
